## 1. Architecture Overview

### Core Classes

| Class | File | Role |
|-------|------|------|
| `Application` | `src/core/Application.h` | Singleton app controller, owns all managers |
| `FileManager` | `src/core/FileManager.h` | Buffer lifecycle, file I/O |
| `Buffer` | `src/core/Buffer.h` | Single document model (text + encoding + lexer) |
| `SessionManager` | `src/core/SessionManager.h` | Save/restore workspace state |
| `MacroManager` | `src/core/MacroManager.h` | Record/playback editor commands |
| `MainWindow` | `src/ui/MainWindow.h` | QMainWindow, owns editor views, menus, docks |

### Command Dispatch (CRITICAL)

**MenuBar → Application → slot**

1. MenuBar emits `menuCommand(const QString& cmd)` signal
2. Application connects: `connect(_menuBar, &MenuBar::menuCommand, this, &Application::onMenuCommand)`
3. `Application::onMenuCommand(cmd)` in `Application.cpp:1166` uses if/else chain mapping:
   - `"file.new"` → `onNewFile()`
   - `"edit.cut"` → `onCut()`
   - `"view.docMap"` → `onToggleDocMap()`
   - etc.
4. Each `onXxx()` is a slot in `Application.h` (line ~328)

**File → Application → slot**

1. `NppCommands` (Scintilla wrapper) emits `command(const QString&)` signal
2. Connected to `Application::onMenuCommand(cmd)` in `Application.cpp:394`

**No separate dispatch table** — all routing is in `Application::onMenuCommand()`.

**Panels (QDockWidget subclasses)** do NOT receive menu commands directly.
All panel actions go through `Application::onMenuCommand()` or the signal bus.

**MainWindow also has commands:**

`MainWindow::dispatchCommand(cmd)` handles window-level commands that don't need `Application`:
- `window.nextView` / `window.prevView` → view cycling
- `window.moveToOtherView` → split view
- `window.toggleSplitVertical`

These are defined in `MainWindow::dispatchCommand()` in `src/ui/MainWindow.cpp`.

**Toolbar consistency**: `toolbar.xml` must use the same command strings as `Application::onMenuCommand()`.
`macro.play` → `macro.playback` was fixed in `ToolBar.cpp` (lines 203, 206, 558, 674) and `MainWindow.cpp` (line 730).

### Buffer System

`BufferID` is a `uint32_t` handle. `BUFFER_INVALID = 0`.
`FileManager::_buffers` is a `std::vector<Buffer*>` — private.
Use `getBufferAt(index)`, `getBufferCount()`, `getBufferPath(id)` etc.

Opening a file:
1. `FileManager::openFile(path)` → creates `Buffer`, adds to `_buffers`
2. Emits `bufferOpened(BufferID)` signal
3. `MainWindow::onBufferOpened()` creates Scintilla tab
   - **IS connected** in `MainWindow::createConnections()` (line 454):
     `connect(&app, &Application::bufferOpened, this, &MainWindow::onBufferOpened);`
   - Slot takes `BufferID` (uint32_t handle), not `Buffer*`

### Panel System

Each panel is a `QDockWidget` subclass. Panels are created in `MainWindow::createPanels()`.
Three panel categories:

1. **Editor panels** (owned by `ScintillaEditView`):
   - `FunctionListPanel` — function/class tree, parses lexer output
   - `DocumentMapPanel` — minimap scrollable editor preview
2. **File panels** (owned by `MainWindow`):
   - `FileBrowserPanel` — file tree for project directories
3. **Tool panels** (owned by `MainWindow`):
   - `ClipboardHistoryPanel` — clipboard ring

**Dock tabification**: Related panels are tabified via:
```cpp
tabifiedDockWidgets(Qt::LeftDockWidgetArea) → [FileBrowserPanel, FunctionListPanel, DocumentMapPanel]
```

**Panel commands** go through `view.fileBrowser`, `view.functionList`, `view.docMap`, etc.
In `Application::onMenuCommand()`:
```cpp
else if (cmd == "view.docMap") { onToggleDocMap(); }
```

`onToggleDocMap()` is a slot that calls `mainWindow()->togglePanel("docMap")`.

## 2. MOC / Qt Meta-Object System

### The Problem

`Application`, `DpiManager`, `ThemeManager`, and all panel classes (`DocumentMapPanel`, `FileBrowserPanel`, `FunctionListPanel`, etc.) have `Q_OBJECT` macros. Without the MOC-generated `moc_*.cpp` files, the `vtable for X` linker errors appear.

### The Solution (in `src/tests/CMakeLists.txt`)

```cmake
# 1. Generate MOC files explicitly (qmake-style, no AUTOMOC needed)
set(MOC_GEN_DIR "${CMAKE_CURRENT_BINARY_DIR}/moc_gen")
function(add_moc HDR OUT_NAME)
    set(MOC_OUT "${MOC_GEN_DIR}/moc_${OUT_NAME}.cpp")
    add_custom_command(OUTPUT "${MOC_OUT}"
        COMMAND Qt6::moc ARGS "${REL_ROOT}/${HDR}" -o "${MOC_OUT}"
        DEPENDS "${REL_ROOT}/${HDR}" COMMENT "Running moc on ${HDR}" VERBATIM)
    list(APPEND MOC_SOURCES "${MOC_OUT}")
    set(MOC_SOURCES "${MOC_SOURCES}" PARENT_SCOPE)
endfunction()

add_moc("core/Application.h" Application)
add_moc("common/DpiManager.h" DpiManager)
add_moc("core/ThemeManager.h" ThemeManager)
add_moc("panels/FileBrowserPanel.h" FileBrowserPanel)
add_moc("panels/DocumentMapPanel.h" DocumentMapPanel)

# 2. Build as SHARED lib (SHARED → full linker processing)
add_library(test_stubs SHARED StubApplication.cpp ${MOC_SOURCES}
    panels/FileBrowserPanel.cpp panels/DocumentMapPanel.cpp)
set_target_properties(test_stubs PROPERTIES AUTOMOC OFF)  # CRITICAL: prevents duplicate MOC

# 3. test_core links test_stubs → all MOC symbols available
target_link_libraries(test_core PUBLIC test_stubs ...)
```

### Key Rules

1. **Panel `.cpp` files MUST be in `test_stubs`** (the SHARED lib with `AUTOMOC OFF`), NOT in `test_core`.
   Rationale: MOC symbols must be defined in the same shared library that exports them.
   When panel `.cpp` files are in `test_core` (STATIC), their MOC symbols are compiled into `test_core`,
   but AUTOMOC also generates MOC into `test_stubs`, causing duplicate-definition linker errors.

2. **`AUTOMOC OFF` on `test_stubs`** is essential. Without it, AUTOMOC re-runs on `FileBrowserPanel.cpp`
   and `DocumentMapPanel.cpp` and generates its own MOC, conflicting with the explicit MOC.

3. **Include `Qt6::Xml` and `Qt6::PrintSupport`** in `test_stubs` because:
   - `FileBrowserPanel.cpp` includes `FunctionListXmlParser.h` → needs `QDomDocument` (Qt6::Xml)
   - `ScintillaEditor.h` needs `QPrinter` (Qt6::PrintSupport)

4. **StubApplication.cpp** provides ALL slot implementations for `Application` and `DpiManager`.
   All 104+ `Application` slot methods must be stubbed (return types: void → empty body, bool → return false, etc.).

### AUTOMOC vs Explicit MOC

- AUTOMOC scans `.cpp` files for `#include "Foo.h"` where `Foo.h` has `Q_OBJECT` → runs `moc Foo.h`.
- Works automatically for SHARED/STATIC libs in most cases.
- **Fails** for transitive includes: `StubApplication.cpp` → `TestStubs.h` → `Application.h`.
  AUTOMOC doesn't see the indirect Q_OBJECT, so it doesn't run moc.
- **Fix**: explicit `add_custom_command` with `Qt6::moc` for the headers we need.

## 3. Testing Strategy

### Test Organization

```
src/tests/
  CMakeLists.txt          — test targets and dependencies
  StubApplication.cpp    — Application/DpiManager stubs (104+ slot implementations)
  TestStubs.h            — Q_OBJECT header include for AUTOMOC detection
  test_*.cpp             — one executable per test file
```

### Test Libraries

- `test_stubs` (SHARED): MOC symbols + Application/DpiManager stubs
- `test_core` (STATIC): core sources WITHOUT Q_OBJECT classes

### Adding a New Test

1. Write `test_foo.cpp` in `src/tests/`
2. Use `#include "core/SomeClass.h"` (no prefix — `REL_ROOT/src/` is in include path)
3. Use `#include <QDebug>` for output (NOT `#include <iostream>` with `std::cerr` for QString!)
4. Use `qFatal()` for assertions, NOT `std::cerr << QString` (won't compile)
5. Add to `CMakeLists.txt`:
   ```cmake
   add_core_test(test_foo test_foo.cpp)   # links test_core + test_stubs
   add_test(NAME FooTests COMMAND test_foo)
   set_tests_properties(FooTests PROPERTIES ENVIRONMENT "QT_QPA_PLATFORM=offscreen")
   ```

### ASSERT Macro Pattern

```cpp
#define ASSERT_TRUE(x) do { if (!(x)) { qFatal("FAILED: %s at %s:%d", #x, __FILE__, __LINE__); } } while(0)
#define ASSERT_FALSE(x) do { if (!!(x)) { qFatal("FAILED: %s at %s:%d", #x, __FILE__, __LINE__); } } while(0)
#define ASSERT_EQ(a, b) do { auto _a = (a); auto _b = (b); \
    if (_a != _b) { qFatal("FAILED: %s != %s at %s:%d", #a, #b, __FILE__, __LINE__); } } while(0)
#define RUN_TEST(name) do { qDebug() << "  RUN " << #name << "..."; \
    name(); qDebug() << "  PASS"; ++g_passed; } while(0)
```

**DO NOT use `std::cerr` with QString** — won't compile.

**DO NOT use `QString::number()` on `QString`** — no such overload.

### Test Stubs

When a class needs `Application` in tests:
```cpp
// In StubApplication.cpp (all slot implementations):
void Application::onToggleDocMap() { }  // empty stub
void Application::onMenuCommand(const QString&) { }  // empty stub
// + 100 more slots from Application.h lines 328-458
```

### StubApplication.cpp Structure

```
1. #include "TestStubs.h"  (includes Application.h, DpiManager.h, ThemeManager.h, FileBrowserPanel.h)
2. DpiManager stub methods (instance(), onScreenChanged(), onScreensChanged())
3. ThemeManager stub methods (instance(), availableThemes())
4. Application singleton + constructor
5. ~104 Application slot stubs (all from Application.h public slots section)
```

## 4. CMakeLists.txt Patterns

### Critical Patterns

```cmake
# ✓ Correct: SHARED + AUTOMOC OFF + explicit MOC
add_library(test_stubs SHARED StubApplication.cpp ${MOC_SOURCES}
    panels/FileBrowserPanel.cpp panels/DocumentMapPanel.cpp)
set_target_properties(test_stubs PROPERTIES AUTOMOC OFF)

# ✓ Correct: test_core links test_stubs (not test_core itself)
target_link_libraries(test_core PUBLIC test_stubs Qt6::Core Qt6::Widgets ...)

# ✗ WRONG: OBJECT library with AUTOMOC — MOC files won't link properly
# ✗ WRONG: AUTOMOC ON with SHARED when we also have explicit MOC → duplicate MOC symbols
# ✗ WRONG: Panel .cpp files in test_core (STATIC) — MOC symbols won't export
```

### Qt Components Required

| Include | Qt Component |
|---------|-------------|
| `<QDomDocument>` | Qt6::Xml |
| `<QPrinter>` | Qt6::PrintSupport |
| `<QNetwork>` | Qt6::Network |
| `<QSql>` | Qt6::Sql |

Add missing components to `target_link_libraries`.

## 5. Common Pitfalls

### 1. `Q_OBJECT` not found by AUTOMOC
**Symptom**: `undefined reference to vtable for Application`
**Fix**: Add explicit `add_moc("core/Application.h" Application)` + compile moc into test_stubs

### 2. Duplicate MOC symbols
**Symptom**: `multiple definition of MapTextView::metaObject() const`
**Fix**: Set `AUTOMOC OFF` on the target that has explicit MOC

### 3. MOC not generated at all (OBJECT library)
**Symptom**: `undefined reference to vtable for DpiManager`
**Fix**: Use SHARED library, not OBJECT. OBJECT libraries don't reliably propagate MOC output.

### 4. `std::cerr << QString` doesn't compile
**Symptom**: `no match for 'operator<<' (operand types are 'std::ostream' and 'QString')`
**Fix**: Use `qDebug()` or `qWarning()` instead. Or use `qUtf8Printable(str)`.

### 5. `QString::number()` called on QString
**Symptom**: `no matching function for call to 'QString::number(QString&)'`
**Fix**: Use `#a` / `#b` (macro stringification) for the name only, no value formatting.

### 6. `namespace ClipboardFormat` conflicts with `enum class ClipboardFormat`
**Symptom**: `redeclared as different kind of entity`
**Fix**: Rename `namespace ClipboardFormat` to `namespace ClipboardMime` in `ClipboardAdapter.h` and update `.cpp` usages.

### 7. Protected method called from test
**Symptom**: `is protected within this context`
**Fix**: Use public API only. E.g., `panel.resize(w, h)` instead of `panel.resizeEvent(&event)`.

### 8. Missing `include_directories` for test
**Symptom**: `Foo.h: No such file or directory`
**Fix**: Use `core/Foo.h` (relative to `REL_ROOT`), not `"core/Foo.h"` from within test sources.
  Or ensure `${TEST_INC}` (which includes `${REL_ROOT}`) covers the path.

### 9. Agent rewrites CMakeLists.txt
**Symptom**: CMakeLists.txt changes unexpectedly between edits
**Fix**: Write the complete file in one `write()` call. The agent can corrupt incremental edits.

### 10. Missing Application slot stubs
**Symptom**: `undefined reference to 'Application::onToggleDocMap()'`
**Fix**: Add stub to `StubApplication.cpp`:
  ```cpp
  void Application::onToggleDocMap() {}
  ```
  Get the exact signature from `Application.h` line ~328+.

## 10. Known Issues

1. **Project panels** — `_projectDock1/2/3` added to MainWindow but no real `ProjectPanel` class exists.
   The `view.projectPanel_1/2/3` actions exist but map to nothing visible.
   The `FileBrowserPanel` is used instead for project file browsing.

2. **DocumentMap scroll sync** — `onEditorScroll()` updates minimap but viewport overlay may need tuning.
   The timer-based `scheduleMinimapUpdate()` approach may need adjustment for large files.

3. **CMakeLists.txt agent corruption** — the agent keeps rewriting CMakeLists.txt
   between edits. Always use `write()` for complete file replacement.
   NEVER use `sed` for multi-line replacements in CMake files.
